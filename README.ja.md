# Prompt Craft

[![License: MIT](https://img.shields.io/badge/License-MIT-green.svg)](https://opensource.org/licenses/MIT)

[English](README.md) | [한국어](README.ko.md) | **日本語** | [中文](README.zh.md)

**Claude Code向けプロンプトレビュー — コードレビューは結果を見る、プロンプトレビューは原因を見る。**

> お互いのコードをレビューしていますよね。プロンプトもレビューしませんか？
>
> Prompt CraftはClaude Codeで使用したプロンプトをキャプチャし、8つの品質基準でスコアリングし、
> 専用レビューリポジトリにGitHub PRを作成します — チームがコード自体ではなく、
> コードを生成した*指示*にフィードバックできるようになります。

---

## 何をするのか

```
いつも通りClaude Codeで作業してください。
追加ステップなし。中断なし。

                                            ┌─────────────────────────────┐
  git push  ──→  メタデータを無音で記録      │  ~/.claude/pushlog.jsonl    │
  git push  ──→  メタデータを無音で記録      │  (ネットワークなし、ブロックなし)│
  git push  ──→  メタデータを無音で記録      └─────────────────────────────┘
                                                        │
  /review   ──→  セッションからプロンプト収集             │
             ──→  8基準でスコアリング         ◄─────────┘
             ──→  レビューリポジトリにPR作成

                        ▼

  ┌─ レビューリポジトリのPR ──────────────────────────────┐
  │                                                      │
  │  PR body       Score badge + Review Focus            │
  │                                                      │
  │  Files Changed:                                      │
  │    prompts.md   ← レビュアーがラインコメントする場所     │
  │    summary.md   ← スコアカード、改善案、メタデータ       │
  │                                                      │
  └──────────────────────────────────────────────────────┘
```

**開発中のフリクションゼロ。** Push hookは無音（<100ms、承認不要）。
重い処理（スコアリング、PR作成）は`/review`実行時のみ。

**プロジェクト無痕跡。** プロジェクトリポジトリには何も書き込みません。すべてのレビューPRは別リポジトリ（例: `my-org/ai-sessions`）に作成されます。

## クイックスタート

**ステップ1: インストール**
```
/install-plugin https://github.com/sdb016/prompt-craft
```

**ステップ2: セットアップ（インストール直後に実行）**
```
/setup           # レビューリポジトリを確認 → すべて自動設定
```

**ステップ3: 使用**
```
# いつも通り作業... コードをpush... 準備ができたら:
/review          # チームフィードバック用プロンプトレビューPR作成
/score           # ローカルスコアリングのみ（PRなし、即時フィードバック）
```

> **注意:** 初回インストール後、push hookを有効にするにはClaude Codeの再起動が必要です。

## なぜPrompt Craftなのか？

| 問題 | Prompt Craftの解決方法 |
|------|----------------------|
| 「Claudeがスコープ外に出た」 | プロンプトレビューが不足している終了基準とスコープ制約を明らかに |
| 「あの人はどうやってClaudeをうまく使ってるの？」 | チームが優れた結果を生んだ正確なプロンプトシーケンスを確認可能 |
| 「プロンプトは動くけど非効率な気がする」 | 8基準スコアリング + 具体的なリライト提案 |
| 「コードレビューは結果しか捉えない」 | プロンプトレビューがコードを生んだ*指示*を捉える |
| 「改善したいけど何を変えればいいかわからない」 | プロンプト別ミニスコアがどのプロンプトを改善すべきか正確に提示 |

## スキル

| スキル | 用途 |
|--------|------|
| `/review` | チームフィードバック用プロンプトレビューPR作成 |
| `/score` | ローカルスコアリング + 改善ヒント（即時、PRなし） |
| `/insights` | スコアトレンド、高スコアパターン、セッション比較 |
| `/prompt-guide` | タスク前プロンプトガイド + 再利用テンプレート |
| `/setup-project` | プロジェクト別キャプチャ設定（有効化、無効化、カスタムリポジトリ） |

### review

| コマンド | 動作 |
|----------|------|
| `/review` | プロンプトレビューPR作成 |
| `/review --status` | 設定 + 最近のPR表示 |
| `/review --doctor` | 前提条件チェック (git, gh, jq) |
| `/review --setup` | 設定再構成 |

### insights

| コマンド | 動作 |
|----------|------|
| `/insights` | 時系列スコアトレンド |
| `/insights --team` | 全著者のトレンド |
| `/insights patterns` | 高スコアプロンプトパターン抽出 |
| `/insights compare #1 #2` | 2つのレビューセッション比較 |

### prompt-guide

| コマンド | 動作 |
|----------|------|
| `/prompt-guide` | コンテキストベースのプロンプト作成ヒント |
| `/prompt-guide template save/list/use/delete` | 再利用テンプレート管理 |

### setup-project

| コマンド | 動作 |
|----------|------|
| `/setup-project` | 現在のプロジェクトキャプチャ状態を表示 |
| `/setup-project on` | 現在のプロジェクトのキャプチャを有効化 |
| `/setup-project on --repo R` | 特定のレビューリポジトリでキャプチャを有効化 |
| `/setup-project off` | キャプチャを無効化（自動スキップ） |
| `/setup-project list` | 全プロジェクト設定を表示 |
| `/setup-project reset` | スキップしたプロジェクトの再質問を有効化 |

## 仕組み

### 開発中（自動、無音）

Claude Code内で`git push`するたびにローカルログファイルに1行を無音で追加します。ネットワークなし、AIなし、ブロックなし — 気づきもしません。

### レビュー時（オンデマンド）

`/review`を実行すると（または`gh pr create`が検出されると）、Prompt Craftが:

1. ローカルログからpushメタデータを読み取り
2. 現在のセッションからすべてのプロンプトを収集
3. 8つの品質基準でスコアリング（単一LLM呼び出し）
4. レビューリポジトリに2つのファイルを作成:
   - **`prompts.md`** — プロンプトシーケンス（プレーンテキスト、レビュアーがラインコメントする場所）
   - **`summary.md`** — スコアカード、コードインパクト、改善提案、メタデータ
5. Files Changedを指す簡潔なPR bodyでPRをオープン

### 自動フロー

| イベント | 動作 | 場所 |
|----------|------|------|
| `git push` | pushメタデータを無音記録 | ローカルJSONL (<100ms) |
| `gh pr create` | プロンプトスコアリング + レビューPR作成 | レビューリポジトリPR |
| `/review`（手動） | 任意のタイミングでレビューPR作成 | レビューリポジトリPR |
| `/score`（手動） | ローカルスコアリング、結果キャッシュ | ローカルのみ（即時） |

## PR構造

レビュアーは**Files Changed**タブで2つのファイルを確認します:

| ファイル | 目的 | レビュアーの行動 |
|----------|------|-----------------|
| **`prompts.md`** | プロンプトシーケンス（プレーンテキスト） | 特定のプロンプトにラインコメント |
| **`summary.md`** | スコアカード、コードインパクト、改善案、メタデータ | コンテキストとスコアリング詳細を確認 |

PR bodyは簡潔な要約です: Score badge + Review Focus + 「Files Changedを開いてレビュー」

## スコアリングシステム

8基準、100点満点。単一LLM呼び出しで評価。

| 基準 | 配点 | 評価内容 |
|------|------|---------|
| 目標明確性 (Goal Clarity) | 20 | 目標が具体的で、結果が意図と一致しているか？ |
| 範囲制御 (Scope Control) | 15 | 境界が設定され、制約が明示されているか？ |
| コンテキスト充足性 (Context Sufficiency) | 15 | 十分な背景が提供されているか？ |
| 終了基準 (Exit Criteria) | 10 | 完了/停止条件が明示され、遵守されているか？ |
| 分解 (Decomposition) | 10 | 複雑なタスクが適切に分割されているか？ |
| 検証戦略 (Verification Strategy) | 10 | 検証方法が明示されているか？ |
| 反復品質 (Iteration Quality) | 10 | フォローアップリクエストが具体的で明確か？ |
| 複雑度適合 (Complexity Fit) | 10 | プロンプトの精緻さがタスクの複雑さに適しているか？ |

**グレードバンド:** 🟢 90–100 Excellent | 🔵 70–89 Good | 🟡 50–69 Needs Work | 🔴 0–49 Poor

### プロンプトギャップモデル

| ギャップ | 意味 | 評価者 |
|----------|------|--------|
| Gap 1: 意図 → プロンプト | プロンプトが実際に望んだことを捉えたか？ | 人間（レビューを通じて） |
| Gap 2: プロンプト → コード | プロンプトの欠陥がコードの欠陥を引き起こしたか？ | AI（原因→結果チェーン） |

## トラブルシューティング

| 問題 | 解決 |
|------|------|
| インストール後Hookが動作しない | Claude Codeセッションを再起動してプラグインhookを再読み込み |
| 設定エラー | `~/.claude/prompt-review.config.json`を削除して`/setup`を実行し再設定 |
| `/review`でプロンプトが表示されない | 作業したセッションから`/review`を実行。または先に`/score`を実行して結果をキャッシュ |
